Adds diagram creation (via Lucidchart) to the Cortex AI track, built entirely
on native Cortex Agent capability â€” no Python `requests` call, no External
Access Integration on a Snowpark proc. This is deliberate: Cortex here has no
outbound network access of its own (see `../CLAUDE.md`), so the external call
to Lucid has to happen through the Agent's own MCP connector, which is a
Snowflake-managed control-plane feature, not a UDF/proc network egress.

`draft_diagram_brief` never talks to Lucid. It only produces the
`description` text that the Agent later hands to Lucid's `create_diagram`
tool. The actual outbound call to Lucid is made by Snowflake's own MCP
connector runtime, authenticated via the `EXTERNAL MCP SERVER` object.

## Prerequisites

1. **Lucid side**: a Lucid workspace admin must enable MCP access in the
   Lucid Admin Panel. Confirm whether your Lucid plan issues a static OAuth
   client id/secret or relies purely on Dynamic Client Registration (DCR) â€”
   the SQL script defaults to DCR per Lucid's public MCP announcement, with
   the OAuth2/static-credential variant commented out as a fallback.
2. **Snowflake side**: a role with privileges to create `API INTEGRATION`,
   `EXTERNAL MCP SERVER`, and `AGENT` objects (typically `ACCOUNTADMIN`, or a
   custom role granted those specific `CREATE` privileges) â€” this is a
   separate, higher-privilege step from deploying the Snowpark proc.
3. A warehouse for the agent's tool execution (referenced as
   `<YOUR_WAREHOUSE>` in the SQL script).

## Deploy

```bash
# 1. Deploy the new stored proc (from snowflake/)
snow snowpark build
snow snowpark deploy

# 2. Run the account-level setup (needs the elevated role above)
snow sql -f snowflake/sql/lucid_mcp_setup.sql
# â€” or paste sql/lucid_mcp_setup.sql into a Snowsight worksheet and fill in
#   <YOUR_WAREHOUSE> (and OAuth client id/secret, if your Lucid plan uses
#   static credentials instead of DCR) before running.

# 3. Run the pure-Python tests
uv run pytest snowflake/tests
```

First real use of `lucid_mcp_server` will trigger a one-time OAuth consent â€”
whoever runs the agent's first Lucid tool call needs to complete that login
against their Lucid account.

## Exercising it

Send the agent a message referencing an approved PRD, e.g.:

> "Draw a diagram of the checkout flow from this PRD: `<PRD text>`"

Expected flow: the agent calls `draft_diagram_brief`, shows you the resulting
brief (diagram type, title, nodes, edges, description), and â€” per the
agent's `instructions.response` block â€” waits for your explicit confirmation
before calling Lucid's `create_diagram` tool. Once confirmed, it returns the
created document's share link.

## Known limitations â€” read before treating this as production-ready

- **The write-gate is prompt-enforced, not code-enforced.** Unlike the Ticket
  Agent's Python-level gate on `create_jira_issue`, the confirm-before-create
  rule here lives entirely in the agent's `instructions` text. A model that
  ignores instructions could call Lucid's `create_diagram` tool without
  waiting for confirmation â€” there's no hard stop in this design today.
- **Lucid's MCP server currently has no Mermaid import** and only limited
  document-edit support (added 2026) â€” diagram layout is generated by
  Lucid's own AI from the description, not pixel-precise or fully
  reproducible. If exact layout ever matters more than speed, the Standard
  Import REST API (JSON-spec based, deterministic) is the alternative worth
  revisiting â€” but that route needs a Snowpark External Access Integration,
  which is the connectivity path this design was built to avoid.
- **MCP tool calls aren't written to Snowflake's query/monitoring tables**
  the way SQL executions are â€” plan for weaker observability into what was
  actually sent to/received from Lucid.
- **External MCP servers aren't verified by Snowflake** â€” you're trusting
  Lucid's endpoint directly; there's no Snowflake-side sandboxing of what a
  connected external MCP server's tools can do beyond the OAuth scope
  granted.
- **First-use OAuth/DCR consent is per-user**, not a one-time service
  account setup â€” expect every new user of this agent to hit a Lucid login
  prompt on their first diagram request.
